# Documentation

本文件定义跨项目通用的文档与注释规则。

## 核心原则

- 修改代码前，应先阅读将要改动区域的注释、docstring 和相关设计文档
- 注释和 docstring 应被视为规格的一部分，而不是可忽略的装饰
- 若文档与代码冲突，以代码和可执行证据为最终真相，但必须同步修正文档

## 注释分层建议

- Module docstring：模块目的、边界、不变量、跨模块协作关系
- Class docstring：职责、生命周期、状态假设、并发假设
- Function docstring：参数、返回值形状、副作用、异常
- Block comments：解释 why、约束、取舍，而不是复述代码在做什么

## 工作流规则

### 开始前

- 阅读即将修改区域的注释和文档，确认现有约束
- 若缺少关键意图、不变量或边界说明，应在实现过程中补齐

### 进行中

- 当你发现约束、做出设计取舍或调整职责边界时，应同步更新相关注释
- 移动、拆分、重命名职责时，应同步更新受影响的 docstring 和说明文档
- 对非显而易见的边界条件和权衡，优先写清 why

### 完成时

- 删除、重写或修复过时注释
- 保持注释简洁准确，避免把注释写成 changelog
- 若项目维护统一文档入口或 docs index，必要时同步更新导航

## 语言与口径

- 注释与文档语言应跟随目标项目既有约定
- 同一仓库内尽量保持一致的写作口径，不要混用多套风格
- 代码注释优先解释约束、意图和取舍，而不是逐行翻译代码

## Required Checks

在提交涉及行为、边界或设计变化的改动前，至少确认：

- 已阅读将修改区域的注释、docstring 和相关设计文档
- 新增约束、边界条件和关键取舍已经同步到注释或文档中
- 过时、失真的注释已经删除或修正
- 若项目维护 docs index 或统一入口，必要导航是否同步更新
