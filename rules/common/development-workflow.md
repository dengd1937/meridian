# Development Workflow

> 本文件补充了功能开发在 Git 操作之前的完整流程。
> 功能实现工作流描述的是一条开发管线：先研究，再规划，再采用测试优先方式实现，最后再进入提交与推送阶段。

## 功能实现工作流

### 0. Research & Reuse

_在开始任何新的实现之前，这一步是强制性的。_

- **先做 GitHub 代码搜索：** 在写任何新代码前，先搜索现有实现、模板和模式，确认是否已经有可复用方案。
- **第二步查官方文档或主文档：** 先查库的官方文档、厂商文档或一手资料，确认 API 行为、用法和版本差异。
- **只有前两步不够时，再扩展到更广泛的网络检索：** 先用代码搜索和一手文档建立事实，再做更宽泛的补充调研。
- **检查包仓库和生态：** 在自己写工具代码前，先查看 npm、PyPI、crates.io 等包仓库，优先使用成熟库，而不是手写重复能力。
- **寻找可改造的成熟实现：** 优先找已经解决了 80% 以上问题的开源项目、模板或实现，再考虑 fork、移植、包装或裁剪。
- **满足需求时，优先采用成熟方案，而不是从零开始重写。**

### 1. Plan First

- 优先使用 `planner` 或等价能力先产出实现计划。
- 编码前先形成规划文档，例如 PRD、架构说明、系统设计、技术方案或任务清单。
- 先识别依赖关系、主要风险和需要提前验证的假设。
- 将实现拆成阶段或可验证的小步，而不是一次性展开大改。

### 2. TDD Approach

- 优先使用 `tdd-guide` 或等价能力推动实现。
- 先写测试，进入 `RED` 状态。
- 运行测试，确认它确实失败。
- 再写实现，使测试进入 `GREEN` 状态。
- 运行测试，确认它通过。
- 再做重构和清理，进入 `IMPROVE` 阶段。
- 最后检查覆盖率，目标不少于 `80%`。

### 3. Commit & Push

- 使用清晰、详细的提交信息。
- 遵循 Conventional Commits 风格。
- 只有在研究、规划和测试完成后，才进入提交和推送阶段。
- 进入交付阶段后，应继续遵循 [git-workflow.md](./git-workflow.md)。
