# Code Review

本文件定义跨项目通用的代码审查基线。

## 核心目标

审查的重点不是复述作者结论，而是回答：

- 当前改动是否存在会阻塞提交或合并的问题
- 本地验证证据是否足够
- 是否存在作者容易忽略的回归风险、架构问题、安全问题或测试缺口

## 何时应加强审查

以下改动默认应提高审查强度：

- 生产代码、测试、脚本、配置、依赖、CI、部署改动
- 会影响行为、接口契约、数据结构、工作流或排障方式的改动
- Bug 修复、跨模块重构、外部系统集成、权限与敏感数据相关改动

纯说明性文档改动可走轻量路径。

## 审查原则

- 尽量使用相对独立的 reviewer 视角，不要把作者会话直接等同于独立审查
- reviewer 应优先查看 diff、关键文件和验证证据，而不是先相信作者结论
- 默认采用 findings-first 输出：先列问题，再给总体结论
- 即使未发现问题，也应说明检查范围、证据和残余风险
- reviewer 默认只审不改；除非用户明确要求，否则不直接实现修复

## Author 在发起审查前应准备

- 任务目标和成功标准
- 变更范围和关键文件
- `git diff --stat`
- `git diff` 或 PR diff
- 本地验证结果摘要
- 已知风险和未覆盖项

材料不全时，reviewer 应指出缺口，而不是自行脑补。

## Author handoff 要求

- 当流程进入独立 reviewer 阶段时，Author 会话应明确提醒使用新会话或隔离上下文执行审查
- 不应把当前 Author 会话直接改口成“现在开始 review”
- 若已有现成 reviewer skill 或 reviewer 协议，应直接指向该入口，而不是临时口头发挥

## Reviewer 的最低检查面

- 正确性：是否真的实现目标，是否引入明显回归
- 架构与规则：是否破坏项目声明的边界、分层或约束
- 安全性：是否引入敏感信息泄露、不安全默认值或危险输入路径
- 测试充分性：验证是否覆盖主要风险，是否缺少关键直接证据

## Findings 处理要求

- 作者应逐条处理阻塞性 findings
- 对无法立即修复的项，应明确记录原因、影响和风险
- 修改后应重新运行相关验证，而不是只回复“已处理”
- 若关键问题仍存在争议或证据不足，应升级到人工判断

## 回写要求

- reviewer 不应只在聊天窗口返回结论，而应优先把审查结果沉淀到 PR、review 系统或团队约定的协作载体
- 回写内容应包含总体结论、检查范围、关键证据和残余风险
- 若平台限制无法提交正式 approve / request changes，至少应通过 comment 明确写出 verdict

## 输出建议

审查输出至少应包含：

- Review mode
- Scope
- Overall assessment
- Findings
- Open Questions
- Residual Risks

推荐 verdict：

- `APPROVE`
- `REQUEST_CHANGES`
- `COMMENT`

## Required Checks

在输出审查结论前，至少确认：

- 已查看 diff、关键文件和验证证据，而不是只复述作者说明
- findings 是否按阻塞性、风险和证据充分度排序
- 若未发现问题，是否说明检查范围、依据和残余风险
- 结论是否已回写到 PR、review 系统或团队约定的协作载体
