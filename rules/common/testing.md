# Testing

本文件定义跨项目通用的测试与验证基线。

## 核心原则

- 测试的目标是证明改动真的完成任务，并尽量降低明显回归风险
- 默认优先补最小必要层级，而不是机械追求“所有改动都跑最重的测试”
- 外部依赖越多，越要把“逻辑正确”和“真实服务可用”拆开验证

## 选择合适层级

- 单元测试：验证单个函数、规则、字段校验、边界条件
- 集成测试：验证模块之间的协作和受控依赖联动
- 端到端测试：验证真实入口到最终结果的主链路
- Smoke Test：验证真实外部依赖在受控环境中的可用性

## 默认策略

- 纯逻辑、解析规则、字段映射、边界条件优先写单元测试
- 模块协作、路由到服务、编排节点协作优先写集成测试
- 关键成功路径、核心 fallback、关键降级路径再考虑少量高价值 E2E
- 外部真实服务可用性优先用 smoke test 或条件触发验证，不要让其阻塞所有本地开发

## 提交前验证要求

- 至少运行与本次改动直接相关的格式化、lint、测试或验证脚本
- Bug 修复应尽量保留“修复前失败 / 修复后通过”的证据
- 接口、数据模型、配置、部署或行为性文档改动，应补充能直接证明行为正确的验证
- 若没有测试，也应至少给出一条可重复的本地验证路径，而不是只靠肉眼判断

## 测试设计要求

- 测试失败时应尽量能快速定位具体规则或具体链路，而不是只给模糊红灯
- 外部响应契约、关键字段映射和降级逻辑应尽量固定为自动化断言
- 若目标项目存在稳定的 fixture、fake service 或 snapshot，应优先复用，而不是重复造轮子
- 若外部依赖只在特定环境可达，应通过环境变量、标记或跳过条件控制，不要让测试在普通开发环境中无意义失败

## 文档与覆盖

- 当项目维护测试分层文档、覆盖矩阵或 smoke 环境说明时，新增测试应同步更新
- 若测试策略发生仓库级变化，应更新统一规则文档，而不是只在单个模块文档中修补

## Required Checks

在声明“已验证”前，至少确认：

- 已运行与本次改动直接相关的格式化、lint、测试或验证脚本
- Bug 修复是否保留了修复前失败和修复后通过的最小证据
- 关键字段映射、外部响应契约或降级逻辑是否有自动化断言覆盖
- 若没有自动化测试，是否提供了可重复的手工验证路径
