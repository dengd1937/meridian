# Security

本文件定义跨项目通用的安全与敏感信息处理基线。

## 敏感信息

- 禁止在代码、日志、测试样例和文档中硬编码真实密钥、Token、密码、邮箱认证信息或其他敏感凭据
- 敏感配置应通过项目约定的 secret / environment / config 机制注入，而不是直接写入仓库
- 示例配置、`.env.example` 或模板文件中只应保留键名、占位值或脱敏样例

## 配置访问

- 业务代码不应分散地直接读取环境变量
- 项目应提供统一的配置入口，并在该入口中处理默认值、优先级和类型转换
- 配置优先级和敏感 / 非敏感配置分类应有清晰约定，并保持文档同步

## 日志与错误处理

- 日志中不得输出敏感字段或完整凭据
- 记录错误时应保留定位问题所需的上下文，但避免把请求体、认证头或原始 secret 直接打到日志里
- 对外部依赖失败、权限失败和安全相关错误，应尽量带上可追踪的错误类型和上下文

## 外部输入与边界

- 外部输入应在边界层尽早校验，避免把未校验的数据直接传播到核心业务逻辑
- 对结构化输入优先使用明确 schema 或模型校验
- 对协议可能演进的输入，应显式说明兼容策略，而不是默认静默吞掉风险

## 升级条件

以下情况不应仅依靠常规自动化流程快速放行，应提高审查强度或请求人工确认：

- 权限、认证、授权逻辑改动
- 涉及敏感数据读写、脱敏或传输
- 外部系统关键链路或高风险自动化操作改动
- 已发现安全隐患，但证据不足以自动判断是否可接受

## Required Checks

在提交任何涉及输入、凭据或外部系统的改动前，至少确认：

- 没有硬编码真实 secret、token、密码或其他敏感凭据
- 外部输入已在边界层完成基本校验，而不是直接传入核心逻辑
- 日志和错误信息不会泄露认证头、请求体敏感字段或完整凭据
- 权限、认证、授权或敏感数据链路变更已提高审查强度
