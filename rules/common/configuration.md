# Configuration

本文件定义跨项目通用的配置访问与配置项演进规则。

## 核心原则

- 业务代码不应分散地直接读取环境变量
- 项目应提供统一的配置入口，并在该入口中处理默认值、优先级、类型转换和来源映射
- 敏感配置与非敏感配置应有明确分类，并在文档中保持一致

## 配置访问

- 业务模块应通过项目约定的 settings / config 对象访问配置
- 直接读取环境变量的逻辑应尽量收敛在配置入口，而不是散落在 Router、Service 或业务模块中
- 如果项目暂时没有统一配置层，新增配置时应优先补配置入口，而不是继续扩散读取方式

## 新增或变更配置项时

新增配置项时，至少应同步检查以下内容：

1. 配置入口
   在统一配置模型或 settings 层中新增字段、默认值和类型
2. 来源映射
   补齐配置文件、环境变量或其他来源到统一配置层的映射
3. 配置文件与模板
   非敏感配置补到默认配置文件，敏感配置补到 `.env.example` 或等价模板
4. 文档与测试
   若影响使用方式、部署方式或运行行为，应同步更新文档；必要时补对应测试

## 推荐分类

- 非敏感配置：功能开关、超时、重试次数、默认阈值
- 敏感配置：API Key、Token、密码、认证凭据

## 推荐优先级

若项目未定义更具体规则，推荐使用：

```text
environment variables > config files > code defaults
```

敏感配置通常应通过环境变量或 secret 机制覆盖，而不是写入仓库默认配置。

## Required Checks

在新增或修改配置项前，至少确认：

- 配置访问是否继续通过统一 settings / config 入口
- 默认值、优先级、类型转换和来源映射是否已显式定义
- `.env.example`、默认配置文件或模板是否完成同步更新
- 受影响的文档、部署说明和必要测试是否一起更新
