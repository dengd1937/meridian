# Patterns

本文件定义跨项目通用的实现模式与结构约束。

这些规则回答“默认应优先采用什么样的结构”，而不是替代目标项目已有的架构设计。

## 核心原则

- 优先沿用仓库中已经被验证的模式，不要为同类问题重复发明第二套结构
- 外部输入、协议适配、模型转换应尽量收敛在边界层，核心逻辑只处理稳定的领域对象
- 业务逻辑应依赖抽象边界，而不是直接耦合底层存储、网络协议或第三方 SDK 细节
- 新模式只有在现有模式明显不适用时才应引入，并应说明收益与迁移成本

## 常见模式

### Boundary First

- 在入口层完成输入校验、鉴权、协议兼容和错误归一化
- 进入核心逻辑前，尽量把数据收敛成稳定的内部模型
- 与外部系统交互的细节应集中在明确的 client / gateway / repository 边界

### Composition Over Duplication

- 优先组合现有模块、脚本和服务，而不是复制粘贴一份平行实现
- 相似流程应抽出共享组件、共享校验或共享错误处理逻辑
- 抽象只有在能明确减少重复和边界混乱时才值得引入

### Stable Interfaces

- 公共接口应尽量保持输入输出稳定，避免把内部重构细节泄露给调用方
- 跨模块契约变更应同步更新调用点、测试和文档，而不是依赖隐式兼容
- 对协议可能演进的边界，应显式定义兼容策略、废弃策略或版本策略

## Required Checks

在引入新模式、新抽象或跨模块重构前，至少确认：

- 当前仓库里是否已有可复用的既有模式
- 新抽象是否真的减少了重复或耦合，而不是只是换了一层名字
- 边界是否更清晰，而不是把输入解析、业务逻辑和外部调用混在一起
- 公共接口、调用方、测试和文档是否会一起演进
