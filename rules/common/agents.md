# Agents

本文件定义跨项目通用的 Agent 协作与执行基线。

## 核心原则

- 先定位任务类型，再读取最相关的规则、设计文档或 skill，不要一次性加载全部上下文
- 优先把仓库文档和可执行证据视为事实来源，避免凭记忆假设实现细节
- 项目越复杂，越应保持“最小充分上下文”，而不是无差别扩展读取范围

## 进入任务前

- 先确认当前任务属于实现、调试、审查、交付、文档还是设计类工作
- 根据任务类型读取对应入口文档、相关规则和必要的 skill
- 若项目提供 `AGENTS.md`、`CLAUDE.md`、设计文档或 docs index，应先从这些入口建立导航

## 执行中的通用要求

- 修改前先阅读将要改动区域的现有注释、测试和设计说明
- 若发现文档与代码不一致，修复代码时应一并更新相关文档
- 只加载完成当前任务所需的最小上下文；需要更多信息时再逐层展开
- 优先复用现有脚本、测试、文档和 skill，不要重复制造平行流程

## 交付类任务

- 用户提到提交、推送、创建 PR、等待 CI、推进合并时，应按完整交付链路理解需求，除非用户明确要求停在某一步
- 当仓库有专门的交付 skill 时，优先交给对应 skill 执行
- 只要项目启用了 PR 工作流或主分支保护，就不应将“用户要求 push”解释为允许直推主分支

## 独立审查协作

- 当流程需要独立 reviewer 时，Author 会话不应直接在同一上下文中充当 reviewer
- 独立审查优先使用新会话、新上下文或明确隔离的 reviewer 路径
- reviewer 默认应直接给出 findings 和 verdict，而不是先改代码

## 项目覆盖优先级

建议遵循以下优先级：

1. 更贴近代码或目录的项目规则
2. 项目级 Agent 入口文档
3. 本仓库语言层规则，例如 `rules/python/`
4. 本仓库通用 `rules/common/`
5. 本仓库的共享 `.agents/skills/`
6. 工具自身默认行为

当更具体、更新更近的规则存在时，应优先服从该规则。

## Required Checks

在开始执行任务前，至少确认：

- 是否已经识别任务类型，并读取了对应入口文档、规则或 skill
- 是否保持最小充分上下文，而不是无差别扫描整个仓库
- 是否把仓库文档和可执行证据作为事实来源，而不是凭记忆假设
- 若进入独立 reviewer 流程，是否切换到隔离上下文而不是在 Author 会话里自审
