# Performance

本文件定义跨项目通用的性能与资源使用基线。

性能规则的目标不是过早优化，而是在关键路径上避免明显低效实现，并要求基于证据做取舍。

## 核心原则

- 先用事实定位瓶颈，再决定是否优化，不要凭直觉重写代码
- 优先优化真实热路径、关键查询、批处理链路和高频接口，而不是所有代码一视同仁
- 若优化会明显增加复杂度，应同步说明收益、代价和回退方式
- 对 I/O、数据库、网络和序列化热点，应优先减少重复调用、无意义拷贝和过度往返

## 默认约束

- 不要在循环中重复执行可提前缓存、批量化或预计算的昂贵操作
- 不要为了局部便利引入显著的 N+1 查询、重复远程调用或全量扫描
- 大对象、长列表和流式数据应优先考虑分批、懒加载或分页处理
- 高并发或高吞吐路径上的日志、校验和序列化成本应保持可控

## 证据要求

- 宣称“性能已优化”时，应提供基准、profile、日志证据或可重复的对比方法
- 若性能风险暂未完全解决，应明确记录已知瓶颈、影响范围和后续处理计划
- 对关键路径的性能回归，不能只依赖主观体感判断是否可接受

## Required Checks

在提交性能敏感改动前，至少确认：

- 是否有证据表明确实存在需要优化的热路径
- 新实现是否减少了远程调用、重复计算、内存占用或序列化开销
- 是否引入了更高的复杂度、缓存失效风险或一致性问题
- 是否补充了最小必要的验证方式来证明收益或避免明显回归
