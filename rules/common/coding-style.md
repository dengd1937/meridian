# Coding Style

本文件定义跨项目通用的编码与实现约束。

这些规则优先回答“默认应该怎么写代码”，而不是替代项目自己的语言规范或框架约束。

## 核心原则

- 优先复用已有模块、工具函数、脚本和文档，不要先写重复实现
- 修改前先阅读相关模块注释、设计文档和现有测试，避免凭印象改代码
- 代码提交时应保持可读、可维护、可验证，而不是仅仅“功能能跑”

## 命名与结构

- 变量、函数、文件名优先使用目标语言的主流命名约定
- 避免模糊命名，如 `data`、`tmp`、`misc`、`helper2`
- 公共接口、跨模块调用入口和扩展点应使用稳定、可理解的命名
- 文件应保持在可维护范围内；若单文件持续膨胀，应优先拆分职责，而不是继续堆叠

## 类型与接口

- 公共函数、公共方法和跨模块边界优先补齐明确的参数与返回类型
- 优先使用现代、简洁、项目已采用的类型表达方式
- 除非处于外部动态边界，否则避免无约束地使用 `Any` 或等价弱类型
- 外部输入、序列化数据和接口载荷优先在边界处做结构化校验，再向内传递

## 配置与依赖边界

- 业务代码不应到处直接读取环境变量、配置文件或 secret
- 配置访问应统一经过项目约定的配置层或 settings 层
- 若项目尚未形成配置入口，新增配置时应先补统一入口，再在业务代码中使用
- 依赖方向应遵守项目已有架构约束；若项目声明了分层边界，不应跨层直接访问低层实现细节

## 日志与可观测性

- 禁止使用 `print` 作为正式代码中的调试与运行输出方式，改用项目约定的 logging 机制
- 错误日志应包含足够定位问题的上下文，例如请求标识、资源标识、错误类型
- 捕获异常时，优先保留堆栈和触发上下文，而不是只记录一句模糊报错
- 日志中不得泄露敏感信息

## 注释与文档同步

- 注释应解释约束、权衡和边界，而不是简单复述代码
- 修改代码时，应同步修复过时注释和相关文档
- 注释与文档语言应跟随目标项目既有约定，避免在同一仓库中混用多套写作口径

## 默认工具约束

- 优先使用项目已选定的格式化、lint 和依赖管理工具
- 如果项目已统一使用 `uv`、`ruff`、`pytest` 一类工具，应沿用，而不是额外引入平行工具链
- 临时验证脚本只有在确有必要时才应创建；若只是一次性排查产物，不应作为长期仓库文件保留

## Required Checks

在提交实现类改动前，至少确认：

- 是否先复用了现有模块、工具函数、脚本和文档，而不是新增平行实现
- 公共接口和跨模块边界是否保持清晰命名、明确类型和稳定契约
- 配置访问、日志记录和异常处理是否遵守项目既有入口与约束
- 是否沿用了项目已有工具链，而不是引入新的格式化、lint 或依赖管理平行方案
