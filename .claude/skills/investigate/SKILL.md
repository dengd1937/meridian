# Investigate

在修复任何 bug 之前，必须先完成根因调查。**调查报告输出之前，禁止修改任何代码。**

## When to Activate

- 遇到 bug 或非预期行为
- 测试意外失败，原因不明
- 生产环境出现异常
- 功能表现与预期不符
- 用户明确要求调查某个问题

## Core Principle

> 没有调查就没有修复权。

猜测根因然后直接改代码，往往只是打表层补丁。系统性调查确保：
- 修复的是真正的根因，而不是症状
- 不引入新的回归
- 后续写的测试能真正防止问题复现

---

## Investigation Workflow

### Phase 1 — 症状描述（禁止查看代码）

用自己的语言精确描述问题，不看代码，只描述现象：

```
预期行为：[用户/系统预期发生什么]
实际行为：[实际发生了什么]
复现步骤：[最短可复现路径]
复现稳定性：[必现 / 偶现(概率) / 特定条件下]
影响范围：[哪些功能/用户/环境受影响]
首次出现：[已知最早出现时间或版本]
```

**门禁**：无法稳定复现的问题，先找到复现条件再继续。

---

### Phase 2 — 假设清单（禁止修改代码）

不看代码，仅凭对系统的理解，列出 3–5 个可能的根因假设：

```
假设 1：[描述] — 可能性：高/中/低
假设 2：[描述] — 可能性：高/中/低
假设 3：[描述] — 可能性：高/中/低
...
```

按可能性从高到低排序。优先验证最可能的假设，不要同时验证多个。

---

### Phase 3 — 逐一验证（只读，不改）

对每个假设，找对应证据来证实或排除：

**证据来源（按优先级）：**
1. 错误日志 / 异常堆栈
2. 失败的测试输出
3. 相关代码路径（只读）
4. git log / git blame（问题引入时间）
5. 配置文件 / 环境变量
6. 外部依赖版本变更

**验证记录格式：**
```
假设 1 验证：
  证据：[找到了什么]
  结论：[证实 / 排除]
  原因：[为什么]
```

用排除法收敛，直到只剩一个假设未被排除。

---

### Phase 4 — 根因确认报告

调查完成后，输出以下报告（在对话中，不写入文件）：

```
## 调查报告

**根因**：[一句话描述真正的问题所在]

**代码位置**：[文件路径:行号]

**触发条件**：[什么情况下会触发]

**为什么之前没被发现**：[测试盲区 / 边界条件 / 近期变更引入]

**影响范围**：[受影响的功能/模块/用户]

**排除的假设**：
- 假设 X：[排除原因]
- 假设 Y：[排除原因]

**建议修复方向**：[不是具体实现，是修复思路]
```

**此时才允许进入修复流程。**

---

## Handoff to Fix

调查报告完成后，根据修复复杂度选择后续路径：

| 情况 | 后续行动 |
|------|----------|
| 根因明确，修复简单（< 10 行） | 直接修复，然后运行 `code-quality-gate` |
| 根因明确，需要写回归测试 | 进入 `tdd-workflow`，将根因作为 RED 阶段的输入 |
| 根因涉及安全问题 | 立即停止，调用 `security-reviewer` agent |
| 根因不明确 | 返回 Phase 2，补充新假设 |

## Common Investigation Mistakes

**错误：边看代码边猜**
- 容易被实现细节带偏，忽略更高层的根因

**错误：找到第一个"可疑点"就停止调查**
- 第一个可疑点往往只是症状，不是根因

**错误：假设和代码修改同步进行**
- 修改代码会破坏现场，导致无法准确判断根因

**错误：跳过复现步骤直接看代码**
- 无法复现意味着无法验证修复是否有效

---

## Pass Criteria

调查完成的判断标准：

- [ ] 问题可以稳定复现
- [ ] 根因已定位到具体代码位置
- [ ] 触发条件已明确
- [ ] 调查报告已输出
- [ ] 后续修复路径已选定
