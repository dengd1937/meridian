# 通用规范

## 代码质量

### 不可变性（CRITICAL）

永远创建新对象，禁止原地修改。不可变数据能避免隐藏的副作用，使调试更容易，并支持安全的并发操作。

### 文件组织

多小文件 > 少大文件：
- 高内聚、低耦合；按功能/领域组织，而非按类型
- 典型 200-400 行，最大 800 行；从大模块中提取工具类

### 错误处理

- 每一层都显式处理错误
- UI 层提供用户友好的错误信息；服务端记录详细上下文日志
- 永远不要静默吞掉错误

### 输入验证

- 在系统边界验证所有用户输入；尽量使用基于 schema 的验证
- 快速失败并给出清晰的错误信息；永远不信任外部数据

### 外科手术式修改

修改现有代码时：
- 每一行改动都应直接对应当前任务
- 发现无关的死代码时，告知用户——不要擅自删除
- 只删除你的改动造成的未使用 import/变量/函数
- 不重构或重新格式化你未被要求修改的相邻代码

### 代码质量清单

标记工作完成前：
- [ ] 代码可读、命名清晰
- [ ] 函数精简（<50 行）
- [ ] 文件聚焦（<800 行）
- [ ] 无深层嵌套（>4 层）
- [ ] 完善的错误处理
- [ ] 无硬编码值（使用常量或配置）
- [ ] 无可变操作（使用不可变模式）

## 架构模式

### 骨架项目

实现新功能时：
1. 搜索成熟的骨架项目
2. 使用并行 agent 评估候选项（安全性、可扩展性、相关性、实现方案）
3. 克隆最优方案作为基础；在成熟结构内迭代

### Repository 模式

将数据访问封装在一致的接口后（findAll, findById, create, update, delete）。业务逻辑依赖抽象接口而非存储机制——便于替换数据源和 mock 测试。

### API 响应格式

所有 API 响应使用统一信封：成功标志、数据载荷（错误时为 null）、错误字段（成功时为 null）、分页元数据（total, page, limit）。

## 安全

### 强制安全检查

任何 commit 前：
- [ ] 无硬编码密钥（API key、密码、token）
- [ ] 所有用户输入已验证
- [ ] 防 SQL 注入（参数化查询）
- [ ] 防 XSS（HTML 已转义）
- [ ] CSRF 保护已启用
- [ ] 认证/授权已验证
- [ ] 所有接口有限流
- [ ] 错误信息不泄露敏感数据

### 密钥管理

- 密钥不得出现在源代码中——由 `pre-write-secrets` hook 在写入时强制执行
- 启动时验证所有必要密钥是否存在；轮换任何可能已暴露的密钥

### 安全响应流程

发现安全问题时：
1. 立即停下
2. 使用 **security-reviewer** agent
3. 修复 CRITICAL 问题后再继续
4. 轮换已暴露的密钥
5. 排查整个代码库中的同类问题

## 测试

### 最低测试覆盖率：80%

必须具备的测试类型：
1. **单元测试** — 独立函数、工具类、组件
2. **集成测试** — API 端点、数据库操作
3. **E2E 测试** — 关键用户流程

### 测试驱动开发

强制工作流：
1. 先写测试（RED）——运行，应当失败
2. 写最小实现（GREEN）——运行，应当通过
3. 重构（IMPROVE）
4. 验证覆盖率（80%+）

### 测试失败排查

1. 检查测试隔离
2. 验证 mock 是否正确
3. 修复实现，而非测试（除非测试本身有误）
4. 卡住时使用 **tdd-guide** agent
