# Planner

> 本 skill 的职责是生成可执行计划，而不是直接写实现代码。目标是让后续执行者在低上下文条件下，仍能按计划稳定推进。

## 角色定位

`planner` 负责回答以下问题：

- 这个任务的范围和边界是什么
- 仓库里有哪些现有实现、模式和规则应优先复用
- 哪些文件、模块、测试、配置会受影响
- 应该按什么顺序实施，才能降低风险并支持增量验证

`planner` 不负责：

- 直接进入编码实现
- 代替 reviewer 做代码审查
- 代替测试 skill 设计具体测试实现细节
- 处理 commit、PR、CI 或交付收口

## 何时使用

以下场景应优先使用本 skill：

- 新功能开发
- 中大型 bug 修复
- 跨文件或跨模块重构
- API、数据模型、配置流、权限边界变更
- 影响面不止一个文件或一个函数的任务
- 需求尚未落成清晰执行步骤的任务

以下场景通常不需要单独使用：

- 单文件小修
- 纯说明性文档修改
- 局部代码解释
- 用户已经提供完整且可直接执行的计划

## 核心原则

1. **Rules First**
- 先读项目规则和仓库约束，再做计划

2. **Reuse Before Rewrite**
- 先找现有实现、现有模式和成熟依赖
- 优先扩展已有结构，而不是为小问题新造抽象

3. **Plan for Low Context**
- 默认后续执行者不了解当前仓库背景
- 计划必须写清精确文件路径、依赖关系和验证方式

4. **Incremental Delivery**
- 计划应支持按阶段推进和局部验证
- 避免“全部做完才能验证”的大爆炸式计划

5. **Testability by Default**
- 每个阶段都应附带对应验证思路
- 高风险改动必须有明确测试或验证安排

## 执行流程

### Step 1: 澄清范围与完成态

先明确：

- 本次任务要解决什么问题
- 哪些内容属于本轮范围
- 哪些内容明确不做
- 什么状态可以视为“完成”

如果需求明显含糊，应先补齐假设和未决问题，而不是直接开始拆步骤。

### Step 2: 研究现有实现与复用点

至少检查以下内容：

- 项目级 `AGENTS.md`、`README`、设计文档、规则文档
- 仓库内相似功能或相似模式的已有实现
- 当前技术栈下可直接复用的模块、工具、依赖或模板

默认策略：

- 先查仓库，再查外部
- 先复用成熟实现，再考虑新建抽象
- 如果复用方案能覆盖需求的 80% 以上，应优先采用

### Step 3: 分析影响面与风险

在拆步骤之前，先锁定：

- 将创建哪些文件
- 将修改哪些文件
- 哪些测试需要新增或调整
- 哪些配置、脚本、文档需要同步更新
- 哪些模块、接口、数据结构会受影响

对高风险点做显式标注，例如：

- 权限或认证链路
- 数据迁移或兼容性
- 外部依赖调用
- 性能敏感路径
- 测试覆盖薄弱区域

### Step 4: 生成可执行计划

计划应满足以下要求：

- 步骤按依赖顺序排列
- 每一步尽量自包含、可验证
- 明确文件路径，不写模糊描述
- 明确验证方式，不只写“补测试”
- 阶段拆分应支持逐步落地

默认优先级：

1. 先搭边界或数据结构
2. 再实现核心行为
3. 再处理异常路径和兼容细节
4. 最后补文档、清理和收口

### Step 5: 输出交接材料

计划完成后，应能直接交给后续实现者继续推进。输出里必须包含：

- 任务概述
- 范围与非范围
- 复用点
- 受影响文件/模块
- 分阶段执行步骤
- 验证策略
- 风险与缓解方案
- 成功标准

如果仍有关键未知项，应在 `Open Questions` 中明确列出，而不是隐含在步骤里。

## 输出格式

默认使用以下结构输出：

```markdown
# Implementation Plan: <task>

## Overview
- 本次要完成什么
- 方案主线是什么

## Scope
- In scope:
- Out of scope:

## Existing Patterns / Reuse
- 现有实现
- 可复用模块
- 需要遵循的规则或模式

## Affected Areas
- Create: `path/to/new_file`
- Modify: `path/to/existing_file`
- Test: `tests/path/to/test_file`
- Docs/Config: `path/to/doc_or_config`

## Implementation Phases

### Phase 1: <name>
1. Step name
- Action:
- Why:
- Dependencies:
- Risk:
- Validation:

### Phase 2: <name>
...

## Testing Strategy
- Unit:
- Integration:
- E2E / workflow:
- Regression focus:

## Risks & Mitigations
- Risk:
  - Mitigation:

## Open Questions
- ...

## Success Criteria
- [ ] ...
- [ ] ...
```

## 计划质量要求

一份合格计划至少满足：

- 写出精确文件路径，而不是笼统写“后端”“前端”
- 每个阶段都可独立验证
- 清楚说明依赖关系和风险
- 不重复设计仓库已有能力
- 不把实现细节写成空泛口号

以下情况说明计划质量不足：

- 没有影响面分析
- 没有验证策略
- 步骤依赖顺序混乱
- 所有任务都堆在一个阶段里
- 计划离开当前作者上下文就无法执行

## 与其他技能的关系

- 方案需要独立评审时，可交给 `design-review`
- 进入具体实现时，由语言或任务相关 skill 接手，例如 `python-patterns`
- 进入测试设计与补测时，可交给 `python-testing`
- 进入提交、PR、CI、merge 阶段后，切换到 `git-workflow`
- 如需适配为工具侧 agent / rule / blueprint，统一遵循根级 `AGENTS.md` 中的 `Tool Adapters` 规范

## 重要约束

- 本 skill 默认只做计划，不直接编码
- 除非用户明确要求，否则不要边规划边实现
- 如果用户已经要求“直接开始做”，也应先在内部完成最小计划，再进入实现
- 当任务实际上是单点小改时，不要强行产出冗长计划

## 交付完成态

以下任一状态可视为 `planner` 阶段完成：

1. 已输出结构化实施计划
2. 已明确范围、依赖、风险和验证方式
3. 已具备可交给后续执行者推进的交接材料

## 推荐用语

开始执行时，可先明确说明：

> 我将先使用 `planner` 生成实施计划，确认影响面、复用点、阶段拆分和验证策略，再进入实现。
