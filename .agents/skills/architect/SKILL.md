# Architect Skill

> 应用架构顾问 — 产出分析和建议，用户拍板。
> 覆盖范围：模块划分、数据模型、API 契约、技术选型、系统集成评估。

## 触发方式

- **手动触发**：用户输入 `/architect`
- **升级触发**：development-workflow 执行中发现架构级问题

## 何时触发

**建议触发**（符合任一条件）：

- 新功能涉及新模块、新数据表、新 API 端点
- 集成外部系统或第三方服务
- 引入新框架、库、中间件
- 数据模型变更（新实体、关系变更、迁移）
- 影响超过 2 个现有模块的改动

**不需要触发**：

- 纯 UI 调整（走 design-workflow）
- 单模块内的小改动
- 配置值变更
- 明显的 lint 修复、typo

---

## 核心原则

1. **顾问角色** — 产出分析和建议，最终决策由用户拍板
2. **基于事实** — 读取现有 ADR、codemap、模块文档后再评估，不凭空推断
3. **方案比选** — 至少提供 2 个候选方案，显式列出优缺点和权衡
4. **决策可追溯** — 每个架构决策产出 ADR，记录上下文、理由和影响

---

## L1 / L2 路由

| 条件 | 级别 | 产出 |
|------|------|------|
| 单模块内改动，技术栈不变，无新依赖 | L1（轻量评估） | mini ADR |
| 跨模块、新依赖、数据模型变更、系统集成、新框架引入 | L2（标准评估） | 完整 ADR |

---

## L1 轻量评估

适用于小范围架构决策。

**流程**：

1. 读取相关的现有 ADR 和模块文档
2. 简要评估影响范围
3. 产出 mini ADR → doc-writer agent 模板：`adr`（状态字段标注为"轻量"）

**Mini ADR**：使用标准 ADR 模板，但备选方案可精简为 1-2 个，影响分析可简要描述。

---

## L2 标准评估

适用于跨模块、引入新技术、数据模型变更等场景。

### 阶段 1：现状理解

**Actions**：

- 读取现有 ADR（`docs/architecture/adr/`），了解已有决策和约束
- 读取 codemap（`docs/CODEMAP.md`），了解项目结构
- 读取相关模块文档（`docs/modules/`），了解现有模块职责和 API
- 读取相关产品文档（`docs/product/`、`docs/designs/`），了解需求上下文
- 调用 **architect agent** 进行深度分析

**目的**：在充分理解现状的基础上评估，避免重复决策或与已有架构冲突。

### 阶段 2：影响评估

**Actions**：

- 分析改动波及的模块和依赖关系
- 评估对数据模型的影响（新实体、关系变更、迁移需求）
- 评估对 API 契约的影响（新增/变更端点、版本策略）
- 评估对现有测试的影响
- 识别风险点（数据迁移、向后兼容、性能瓶颈）

**产出**：结构化影响分析报告。

### 阶段 3：方案比选

**Actions**：

- 基于影响评估，设计至少 2 个候选方案
- 每个方案包含：描述、优点、缺点、对现有系统的影响、实现复杂度
- 给出推荐方案和理由

**产出**：方案比选报告，展示给用户审批。

**Gate**：用户确认方向。

### 阶段 4：决策记录

**Actions**：

- 基于用户确认的方案，产出 ADR
- 传递给 → doc-writer agent 模板：`adr`
- 传递给 → doc-updater agent 更新 ADR Index

**产出**：`docs/architecture/adr/<NNNN>-<slug>.md`

### 阶段 5：交付

**Actions**：

- ADR 作为 planner agent 的输入约束
- ADR 作为 development-workflow 的架构参考
- 如有需要，更新 codemap 和模块文档

---

## 可重入

以下情况可重新触发 architect skill：

- **development-workflow 执行中发现架构级问题** → 停下，执行 `/architect`
- **需求变更导致架构方案需调整** → 更新或新建 ADR
- **集成评估后发现方案不可行** → 回到阶段 3 重新比选

## 与其他工作流的衔接

| 上游工作流 | 输入 |
|-----------|------|
| ideate | 产品定义、功能清单、技术约束 |
| design-workflow | 设计意图、组件契约（如有 UI 相关的架构影响） |
| development-workflow（升级） | 调研中发现的新模块/新依赖/数据模型变更 |

| 下游工作流 | 输出 |
|-----------|------|
| planner agent | ADR 作为实现方案的输入约束 |
| development-workflow | ADR 指导编码方向 |
| doc-updater agent | ADR Index 更新 |
