# TDD Guide

> 本 skill 的职责是用测试驱动实现过程，而不是在代码写完后补几条测试。目标是让实现、回归和重构都围绕可验证的失败测试推进。

## 角色定位

`tdd-guide` 负责回答以下问题：

- 当前这一步应该先写什么测试
- 测试是否真的因为目标行为缺失而失败
- 最小实现应该写到什么程度才足够
- 何时可以进入重构，而不是继续堆功能
- 哪些边界条件和错误路径需要在本轮补上

`tdd-guide` 不负责：

- 代替 `planner` 做范围澄清和实施拆分
- 代替语言层测试 skill 讲解框架细节或 fixture 用法
- 代替 reviewer 做代码审查
- 处理 commit、PR、CI 或 merge 收口

## 何时使用

以下场景应优先使用本 skill：

- 实现新功能
- 修复 bug
- 调整已有行为
- 对既有实现做可观察行为上的重构
- 为缺少回归保护的改动补测试并推进实现

以下场景通常不需要单独使用：

- 纯文档改动
- 只改配置且没有可验证行为变化
- 探索性原型或一次性脚本
- 用户明确要求跳过 TDD，并接受对应风险

## 核心原则

1. **No Production Code Without a Failing Test First**
- 没有先看到失败测试，就不要写生产代码
- 测试如果一开始就通过，说明它没有证明任何新行为

2. **Verify the Failure**
- `RED` 不是“测试跑红了”就行
- 必须确认它因为目标行为缺失而失败，而不是因为拼写、环境、fixture 或断言错误

3. **Minimal Green**
- `GREEN` 阶段只写最小实现
- 不顺手加额外能力，不提前抽象，不借机做无关优化

4. **Refactor Only on Green**
- 只有测试稳定通过后，才进入清理、重命名、提取重复、调整结构
- 重构期间测试必须保持绿色

5. **Behavior Over Implementation Details**
- 测试应描述外部可观察行为，而不是内部实现细节
- 除非无法避免，否则不要把 mock 行为当成真实行为来验证

6. **Evidence Over Claims**
- 任何“已经修好”“应该没问题”都不算完成
- 必须有失败测试、通过测试和必要回归作为证据

## 铁律

```text
没有先失败的测试，就不要写生产代码。
```

如果已经先写了实现代码：

- 不要把它当“参考实现”继续补测试
- 不要一边保留它一边假装在做 TDD
- 默认应回到测试，重新从失败用例推进实现

如确需例外，应由人明确授权，而不是由 agent 自行放宽标准。

## 执行流程

### Step 1: 明确当前这一小步的行为

一次只推进一个清晰行为：

- 这个测试要证明什么
- 通过后外部会观察到什么变化
- 哪些边界条件必须在本轮纳入

如果一个测试名里出现多个并列行为，通常说明粒度太大，应先拆分。

### Step 2: 先写失败测试（RED）

测试要求：

- 名称直接表达行为
- 尽量只验证一个核心点
- 断言具体，不写“跑通即可”
- 优先测试真实代码路径，而不是只测 mock 自己

在写任何生产代码前，先运行测试并确认：

- 测试确实失败
- 失败原因符合预期
- 失败说明了能力缺失，而不是测试写错

### Step 3: 写最小实现（GREEN）

只写让当前测试通过所必需的最少代码：

- 不顺手补未来需求
- 不提前做大规模抽象
- 不把重构、优化、扩展混进这一步

如果需要大量 mock 才能让测试成立，先反查设计是否过度耦合。

### Step 4: 再次运行并确认通过

至少确认：

- 当前新增测试通过
- 相关回归测试仍然通过
- 输出干净，没有新的 warning、error 或明显副作用

如果本轮是 bug 修复，还应保留这个失败用例作为回归保护。

### Step 5: 重构与清理（REFACTOR）

只有在绿色状态下才允许：

- 去重
- 改善命名
- 提取帮助函数
- 收敛接口
- 调整结构以提高可读性和可测试性

重构后必须再次运行测试，确认仍然是绿色。

### Step 6: 补边界与错误路径

主路径通过后，继续检查是否需要补以下测试：

- 空值、空字符串、空数组
- 非法类型或格式错误输入
- 边界值
- 异常路径
- 依赖失败
- 并发或竞态条件
- 大数据量或性能敏感输入

并非每轮都要覆盖全部项，但必须显式判断，而不是默认忽略。

## 输出与交接

使用本 skill 推进实现时，默认应产出以下信息：

- 当前正在实现的行为
- 对应失败测试
- 当前最小实现范围
- 已验证通过的测试
- 尚未覆盖的边界或下一步测试点

如果需要交接给其他 skill：

- 复杂范围或大步拆分先回到 `planner`
- Python 测试细节、fixture、mock、参数化、覆盖率落地交给 `python-testing`
- 代码完成后进入 `code-review-expert`
- 提交、PR、CI 阶段切换到 `git-workflow`

## 质量要求

一轮合格的 TDD 至少满足：

- 测试先于生产代码出现
- 亲眼看到测试因正确原因失败
- 只写最小实现让测试变绿
- 重构发生在绿色状态之后
- 当前行为有自动化回归保护
- 关键边界和错误路径经过显式判断

以下情况说明已经偏离 TDD：

- 代码先写完，再“补几条测试”
- 测试第一次运行就直接通过
- 测试失败原因和目标行为无关
- 为了让测试通过而修改断言掩盖问题
- 为一个小行为顺手实现多个额外特性
- 重构和新功能混在同一步里推进

## 与其他技能的关系

- 复杂任务先用 `planner` 拆出阶段，再由本 skill 推进每个实现小步
- Python 项目里与测试框架、fixture、mock、coverage 相关的具体做法交给 `python-testing`
- 代码改动完成后，优先交给 `code-review-expert`
- 如需适配为工具侧 agent / rule / blueprint，统一遵循根级 `AGENTS.md` 中的 `Tool Adapters` 规范

## 重要约束

- 默认采用 `RED -> GREEN -> REFACTOR`
- 不要把“测试之后补上”称为 TDD
- 不要在没有失败测试的情况下直接修 bug
- 当测试难以编写时，优先怀疑接口或设计过于复杂
- 若用户明确要求跳过 TDD，应在输出中显式说明风险

## 交付完成态

以下任一状态可视为本 skill 当前轮次完成：

1. 已为当前行为写出失败测试，并确认其因正确原因失败
2. 已以最小实现让测试变绿，并保留回归保护
3. 已完成本轮重构，且相关测试保持绿色
